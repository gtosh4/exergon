## Core Market Position

Exergon's strongest lane is not "another space factory game." That territory is already crowded by games that own specific fantasies: Factorio owns logistics mastery, Satisfactory owns embodied 3D factory building, Dyson Sphere Program owns cosmic-scale megastructure spectacle, and Outworld Station occupies accessible space-station automation with co-op and shipbuilding.

Exergon's sharper position is:

> A run-based factory science game where every planet changes the rules, the production graph must be discovered, and each escape proves the player understood the world.

The market notes repeatedly support this direction:

- Run-based builders can work when each run captures the high-agency problem-solving phase and ends cleanly. See [Against the Storm](./against-the-storm.md).
- Factory players reward deep systems but punish friction, opacity, and waiting. See [Factorio](./factorio.md), [shapez](./shapez.md), [Outworld Station](./outworld-station.md), and [GTNH](./gtnh.md).
- Players respond strongly to visible production payoffs and large constructed objects. See [Dyson Sphere Program](./dyson-sphere-program.md), [Satisfactory](./satisfactory.md), and [Outworld Station](./outworld-station.md).
- Knowledge and discovery can be the progression system if the world has coherent rules and strong clue organization. See [Outer Wilds](./outer-wilds.md) and [Duskers](./duskers.md).

## Product Lessons

### 1. The Graph Discovery Hook Must Be Immediate

The first playable experience needs to show that Exergon is not a conventional known-recipe factory game. The player should observe a world property, form a hypothesis, scout or analyze, reveal partial production knowledge, and build differently because of it.

References:

- [Outworld Station](./outworld-station.md): negative reviews warn against shallow known-recipe repetition.
- [Outer Wilds](./outer-wilds.md): validates understanding as progression.
- [GTNH](./gtnh.md): validates deep graph mastery, but also warns against wiki dependency.
- [Against the Storm](./against-the-storm.md): shows that readable run constraints drive replayable adaptation.

### 2. Difficulty Must Come From Decisions, Not Waiting

Across factory references, players are hostile to idle production gates, padded material costs, and repeated manual setup after the problem is already solved. Exergon's difficulty should come from research tradeoffs, uncertain recipes, power transitions, exploration-gated alternatives, and changing world constraints.

References:

- [Outworld Station](./outworld-station.md): late-game waiting and shallow scaling are recurring complaints.
- [Factorio](./factorio.md): rote work becomes automation.
- [shapez](./shapez.md): challenge comes from design complexity, not punishment.
- [GTNH](./gtnh.md): accomplishment is valuable, but grind narrows the audience.

### 3. Planning Tools Are Core Gameplay

The more Exergon depends on seeded graphs and partial information, the more important its tools become. The graph viewer, codex, ratio calculator, bottleneck analysis, diagnostics, ghost planning, and blueprints are not polish. They are how the player thinks.

References:

- [Factorio](./factorio.md): mature tools earn veteran trust.
- [shapez](./shapez.md): pure factory design succeeds through fast iteration and clear tooling.
- [Outworld Station](./outworld-station.md): UI/QoL friction appears repeatedly in complaints.
- [Captain of Industry](./captain-of-industry.md): complex systems need root-cause diagnostics.
- [Outer Wilds](./outer-wilds.md): clue organization is essential for knowledge-gated play.

### 4. Every Run Needs A Complete Arc

The run should not feel like an endless factory that stopped early. It needs landing, orientation, discovery, production, a late synthesis problem, and a dramatic escape. Bad runs should still produce stories and completions, not dead saves.

References:

- [Against the Storm](./against-the-storm.md): the reset works because each settlement has a complete arc.
- [Outworld Station](./outworld-station.md): visible ship launches give production purpose.
- [Dyson Sphere Program](./dyson-sphere-program.md): long goals are stronger when physically visible.
- [Opus Magnum](./opus-magnum.md): valid messy solutions should still feel worth celebrating.

### 5. The Escape Artifact Must Be A Physical Payoff

Factory players notice spectacle when it is earned by production. Exergon's escape artifacts should be visible landmarks, assembled from recognizable components, changed by factory progress, and activated through an in-world event.

References:

- [Dyson Sphere Program](./dyson-sphere-program.md): cosmic-scale construction creates wonder.
- [Satisfactory](./satisfactory.md): large structures and embodied scale create attachment.
- [Outworld Station](./outworld-station.md): ship construction and launches are praised.
- [Outer Wilds](./outer-wilds.md): the final act should use accumulated understanding.

### 6. Planet Identity Must Be Readable, Not Just Procedural

Procedural variance is only valuable when players can reason about it. Planet properties, biomes, resource geography, power constraints, and alien science should combine into worlds players can name and remember.

References:

- [Factorio](./factorio.md): Space Age validates authored worlds with distinct rules.
- [Against the Storm](./against-the-storm.md): biomes and modifiers work because they are legible.
- [Satisfactory](./satisfactory.md): memorable locations create attachment.
- [Outer Wilds](./outer-wilds.md): coherent world rules make discovery satisfying.
- [Dyson Sphere Program](./dyson-sphere-program.md): planets can blur if logistics abstraction erases identity.

### 7. Remote Exploration Must Feed Factory Decisions

Drone scouting should not be free camera movement or detached exploration. It should be risky, information-rich, and directly tied to recipe reveals, unlock vectors, resource routing, and power/biome decisions.

References:

- [Duskers](./duskers.md): limited information and drone risk create tension.
- [Pacific Drive](./pacific-drive.md): excursions work when preparation, danger, return, and upgrade form a loop.
- [Techtonica](./techtonica.md): factory plus exploration can work, but pacing must be precise.
- [Outer Wilds](./outer-wilds.md): physical discovery is stronger than menu discovery.

### 8. Avoid Hidden Death Spirals

Exergon can create pressure through power, research scarcity, reactivity, outpost risk, and drone loss, but every major failure mode needs clear diagnosis and recovery. A bad run should produce a slow or ugly escape, not an unrecoverable collapse.

References:

- [Captain of Industry](./captain-of-industry.md): interlocked systems can collapse into opaque death spirals.
- [Factorio](./factorio.md): players need recoverable first designs.
- [Duskers](./duskers.md): drone loss should sting but not invalidate the campaign structure.
- [Against the Storm](./against-the-storm.md): pressure works when the broader arc continues.

### 9. 3D Presence Is Valuable, But Placement Tedium Is Dangerous

Satisfactory and Infinifactory show that physical presence and 3D construction can create attachment and clarity. They also show that camera, placement, snapping, and large-scale construction friction can become major problems.

References:

- [Satisfactory](./satisfactory.md): embodied scale is powerful, but late-game construction can become tedious.
- [Infinifactory](./infinifactory.md): 3D factories need fast visual testing and excellent placement.
- [Techtonica](./techtonica.md): 3D terrain and factory scale raise performance and pacing risks.

### 10. Combat Is Not The Best Fit For Exergon's Core

Combat helps some factory games create pressure, but it divides the audience. Exergon's current direction, world reactivity, environmental hazards, and drone risk, is better aligned with science discovery and planning.

References:

- [Factorio](./factorio.md): enemies are systemic but still divisive.
- [Dyson Sphere Program](./dyson-sphere-program.md): combat changes the tone of a previously chill automation game.
- [Outworld Station](./outworld-station.md): combat is mixed and can feel underdeveloped.
- [Techtonica](./techtonica.md): factory players are sensitive to side systems that slow the main loop.

### 11. Community Knowledge Should Help Without Solving The Run

Factory games thrive on shared knowledge, blueprints, guides, and optimization culture. Exergon should support seed reports, graph screenshots, codex ranges, run summaries, and blueprints while preserving run-specific uncertainty.

References:

- [Factorio](./factorio.md): modding, blueprints, and guides extend longevity.
- [GTNH](./gtnh.md): community knowledge is central, but external dependency can alienate.
- [Opus Magnum](./opus-magnum.md): shareable solutions create social value.
- [Infinifactory](./infinifactory.md): workshop support extends puzzle life.
- [Outer Wilds](./outer-wilds.md): spoiler collapse is the core risk for knowledge games.

### 12. Performance Budgets Are Design Budgets

Large 3D factories, networks, drones, graph analysis, reactivity simulation, and visible escape artifacts must be scoped against performance early. Performance failure undermines late-game satisfaction.

References:

- [Outworld Station](./outworld-station.md): late-game performance and stability are player complaints.
- [Dyson Sphere Program](./dyson-sphere-program.md): scale fantasies create simulation/rendering pressure.
- [Satisfactory](./satisfactory.md): 3D factory scale requires strong tooling and performance.
- [Techtonica](./techtonica.md): terrain-heavy 3D factory worlds are especially risky.

## Store And Demo Lessons

The demo should prove the differentiator in one short arc:

1. Land on a planet with readable physical properties.
2. Scout with a drone outside the safe field.
3. Collect or observe something that partially reveals a recipe, node, or power constraint.
4. Use the graph/planner to make a decision.
5. Build a compact factory that reflects that decision.
6. Show a visible escape-object or tier-transition payoff.

References:

- [Against the Storm](./against-the-storm.md): show the run arc.
- [shapez](./shapez.md): show planning and tooling proudly.
- [Duskers](./duskers.md): show remote tension.
- [Dyson Sphere Program](./dyson-sphere-program.md): show visible payoff.
- [Outworld Station](./outworld-station.md): avoid generic space-factory positioning.

The store page should not lead with generic phrasing like "build a factory on an alien planet." That places Exergon too close to broad automation competitors. Better hooks:

- "Decode a new alien world's physics each run, reveal its hidden production graph, and build the one machine that can get you home."
- "A run-based factory science game where every planet changes the rules."
- "The depth of an expert factory pack, rebuilt around readable science, modern tools, and unspoilable runs."

## Design Watchlist

Use these as recurring checks during vertical slice and demo development:

- Does the first hour produce a genuine "I figured this world out" moment?
- Can a player understand why a recipe, machine, network, or power system is blocked?
- Are repeated actions compressed by tools before they become chores?
- Does exploration immediately affect production planning?
- Does a bad plan create recovery work rather than a dead run?
- Can screenshots communicate the planet's identity and the escape goal?
- Is the graph tool good enough to be shown in the trailer?
- Does the Standard run avoid idle waiting after the player solves a problem?
- Are performance budgets defined for machine count, network simulation, drones, reactivity, and graph analysis?
- What can players share after a run: seed, graph, base screenshot, escape replay, metrics, or story?

## Consolidated Open Questions

### Positioning

- What is the one-sentence public pitch that does not sound like Factorio, Satisfactory, Dyson Sphere Program, or Outworld Station?
- Which phrase should be avoided because it creates the wrong expectation: "space factory," "survival crafting," "roguelite factory," "automation sandbox," or something else?
- Is GTNH mentioned publicly at all, or kept strictly as an internal depth benchmark?
- Does the product page lead with run structure, science discovery, or factory depth?
- What visual proof shows that this is not a generic alien factory game?

### First Hour

- What exact event creates the first "I figured this world out" moment?
- What does the player know at landing, and what must be discovered?
- Which system is the first seeded variable the player acts on: power, recipe parameters, resource geography, biome effects, or unlock vectors?
- How many recipe/tech nodes should be visible before the first factory is placed?
- What is the first thing the player can get wrong, and how does the game make recovery obvious?

### Run Structure

- What is the minimum viable run arc for the vertical slice?
- What is the shortest satisfying escape condition?
- How does a run escalate without adding idle production time?
- What marks the midpoint of a Standard run?
- What makes an ugly escape different from a clean escape besides completion time?
- Which metrics are shown at run completion, and which are hidden to avoid optimization anxiety?

### Science And Graph Discovery

- What is the minimum graph size that feels deep without overwhelming the first-run player?
- Which recipe facts can be partially revealed, and which require full research?
- What does the player do when a partially revealed recipe has a broad parameter range?
- How does the codex record type-level knowledge without revealing run-specific answers?
- How does the game prevent external wikis from outperforming in-game tools?
- Can a player intentionally pursue an explorer-first or factory-first unlock route in the same seed?

### Tools And UI

- What is the MVP feature set for the graph viewer?
- What is the MVP feature set for the ratio calculator?
- What is the MVP feature set for bottleneck diagnostics?
- Which player actions need copy/paste support?
- Which repeated workflows need explicit click-count targets?
- How does the UI distinguish "unknown," "partially known," and "fully known" without clutter?
- Can a player understand why a machine stopped within five seconds?
- Can a player understand why a tech node is unavailable within five seconds?

### Exploration And Drones

- What information can only be gained by physically scouting?
- What information can be guessed from planet properties before scouting?
- What is the cost of losing a drone in the first hour, Standard runs, and Advanced runs?
- What makes Remote mode tense without making controls slow?
- How long should a typical sample expedition take?
- What does a drone bring back that immediately changes the graph, factory, or power plan?
- How many scouting trips occur before a player can complete Initiation?

### World Reactivity

- What is the first visible sign that the world is reacting?
- What player choice causes it?
- What opportunity can reactivity create, not just what problem?
- How does the game show cause and effect without a hidden pollution-style meter?
- What recovery actions reduce or redirect reactivity?
- Which reactivity systems are excluded from the vertical slice?

### Factory And Scale

- How big is a Standard-run factory in machine count, outpost count, and network size?
- What factory scale should screenshots sell?
- What building operations become automated by Engineering research?
- How does blueprint reuse work when recipe ratios vary per run?
- How does the logistics network stay legible without physical belts?
- How are channel limits taught without feeling arbitrary?

### Power

- What is the first power transition, and why is it interesting?
- Which power rules are stable across all runs?
- Which power rules are seeded?
- How does the UI distinguish voltage failure from amperage failure?
- What happens when an outpost loses power?
- How much punishment is default, and what is moved to challenge modifiers?

### Performance And Technical Scope

- What are the target budgets for active machines, stored items, network edges, drones, outposts, visible cables, graph nodes, and reactivity events?
- Which simulation systems run continuously, and which can be event-driven?
- What is the acceptable save/load time for a mature Standard run?
- What is the target frame rate and minimum hardware profile for late Standard?
- What visual scale is promised in screenshots, and can the engine support it?

### Community And Sharing

- What can players share after a run?
- Is seed sharing a first-class UI feature?
- Can graph snapshots be exported?
- Can blueprints be shared if seeded recipe parameters differ?
- What is Exergon's equivalent of an Opus Magnum GIF or Factorio blueprint?
- What community knowledge should the codex deliberately absorb so external guides are not mandatory?

## Action Items

### Near-Term Design

- Define the vertical-slice run arc from landing to escape, including the first science inference, first factory decision, first drone expedition, and final payoff.
- Write a one-page "first hour" beat sheet showing exactly what the player sees, learns, builds, and misunderstands.
- Define the minimum graph/research model for Initiation, including visible nodes, partial reveals, full reveals, and research currencies.
- Specify the first three planet identities the game can generate, with both mechanical modifiers and screenshot-readable visual traits.
- Decide which systems are excluded from the vertical slice to protect focus.

### UX Prototypes

- Prototype the graph viewer before adding more content.
- Prototype partial recipe reveal states with uncertain ranges.
- Prototype root-cause machine diagnostics for missing input, power, network capacity, and unknown recipe state.
- Prototype the ratio calculator on a seeded recipe chain.
- Prototype ghost planning and one reusable blueprint workflow.
- Time the number of clicks and view changes required to go from recipe discovery to planned factory.

### Gameplay Prototypes

- Build one drone scouting loop where the player risks a drone to reveal production-relevant information.
- Build one power transition that requires a real decision but cannot destroy the run.
- Build one world-reactivity event that creates both a cost and an opportunity.
- Build one escape artifact sequence that changes visually as components are delivered.
- Build one ugly-but-valid escape and one cleaner optimized escape to compare emotional payoff.

### Technical Planning

- Create explicit performance budgets for Standard-run scale.
- Define which simulation systems are deterministic and seed-driven.
- Define run-summary data structures early: seed, planet properties, graph composition, completed escape, metrics, and player notes.
- Define content schema boundaries for materials, forms, recipes, tech nodes, planet modifiers, biomes, drones, power sources, and escape conditions.
- Add automated validation requirements for generated runs: solvability, graph reachability, recipe bounds, power availability, and resource placement.

### Market And Demo

- Draft three alternate Steam short descriptions and test which one communicates the differentiator fastest.
- Build a screenshot checklist: planet identity, drone scouting, graph planning, compact factory, outpost, escape artifact.
- Ensure the trailer shows the graph tool as a selling point, not a background UI.
- Avoid leading with generic "build a factory on an alien planet" language.
- Decide whether co-op is explicitly out of scope for launch or preserved as a long-term possibility.

### Playtest Questions

- Did players understand what was seeded and what was stable?
- Did players form a correct hypothesis from planet properties before being told what to do?
- Did players use the graph tool voluntarily?
- Did players know why production stopped?
- Did players feel exploration changed their factory plan?
- Did players wait after solving the problem, or did the game keep presenting decisions?
- Did players describe the run by character, such as "low-oxygen geothermal world," or only by mechanics?
- Did players want to start a second run for a meaningfully different problem?
